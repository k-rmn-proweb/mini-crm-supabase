## 1. Контекст и цель проекта

### Зачем этот проект существует

Это **портфолио-проект** frontend-разработчика (15 лет опыта, специализация React/TypeScript), который выходит на фриланс-биржу Upwork. У разработчика есть глубокий коммерческий опыт, но нет **публичных, кликабельных ссылок** на работы (все реальные проекты — за корпоративным логином и не могут быть показаны заказчику).

Этот проект решает три задачи одновременно:

1. **Живая ссылка для откликов.** Заказчики на Upwork регулярно требуют «share a link to a live dashboard/app you've built». Нужен работающий публичный URL, который заказчик может открыть и потыкать своими руками.
2. **Доказательство владения стеком.** Проект использует ровно тот стек, который ищут топовые клиенты (React + TS + Supabase). Он должен визуально и функционально доказывать: «этот человек умеет строить production-grade приложения на этом стеке».
3. **Освоение Supabase на практике.** Разработчик силён во фронтенде и знает PostgreSQL, но Supabase как платформу осваивает впервые. Проект должен пройти всю механику Supabase (auth, база, RLS, запросы) руками.

### Как это влияет на технические решения

Поскольку это **витрина для заказчика, а не учебная поделка**, приоритеты такие:

- **Качество важнее количества фич.** Лучше 4 экрана, выглядящих как настоящий продукт, чем 15 сырых. Заказчик судит по первому впечатлению от живого URL.
- **Чистый, профессиональный UI.** Это должно выглядеть как реальный внутренний бизнес-инструмент (типа HubSpot/Pipedrive в миниатюре), а не как студенческий проект. Аккуратная типографика, консистентные отступы, продуманные состояния (loading/empty/error).
- **Production-grade код.** Строгий TypeScript, обработка ошибок, никаких `any`, никаких `TODO later`, никаких полусделанных фич. Код может смотреть заказчик-разработчик — он должен быть чистым.
- **Реалистичность данных.** Seed-данные должны выглядеть правдоподобно (реальные имена компаний, суммы сделок), чтобы дашборд смотрелся живым, а не пустым.

### Что это НЕ

- Не реальный продукт для продакшена с платящими пользователями.
- Не требует масштабирования, сложной инфраструктуры, микросервисов.
- Не требует покрытия тестами на 100% (базовые проверки достаточно).
- Всё делается на **бесплатных тарифах** Supabase (Free) и Vercel (Hobby).

## 2. Продукт в одном предложении

**Mini-CRM** — веб-приложение для управления клиентами и сделками с аналитическим дашбордом: пользователь регистрируется, ведёт базу своих клиентов, отслеживает сделки по этапам (воронка), записывает активности (звонки/письма/встречи) и видит сводную аналитику на дашборде.

Аналогия для понимания: упрощённый Pipedrive / HubSpot CRM для одного пользователя.

## 3. Технологический стек (обязательный)

Стек выбран намеренно, чтобы совпадать с требованиями топовых заказчиков. Не заменять компоненты без веской причины.

| Слой | Технология | Обоснование |
|---|---|---|
| Сборка | **Vite** | Быстрый dev, стандарт для React SPA |
| UI-фреймворк | **React 19** (строго) / **TypeScript (strict mode)** | Ядро проекта. Именно 19 версия — это требование топовых заказчиков |
| Стили | **Tailwind CSS** | Стандарт современного фронтенда |
| UI-компоненты | **shadcn/ui** | Готовые качественные компоненты (кнопки, диалоги, таблицы, формы) |
| Роутинг | **TanStack Router** | Типобезопасный роутинг с генерацией дерева маршрутов. Сильнее и современнее React Router; редкий навык — плюс к профилю |
| Серверное состояние | **TanStack Query (React Query)** | Загрузка, кеширование, мутации данных из Supabase |
| Клиентское состояние | **Zustand** | UI-состояние (сайдбар, фильтры, тема, модалки). НЕ для серверных данных |
| Формы | **React Hook Form + Zod** | Валидируемые формы (создание/редактирование клиентов и сделок) |
| Бэкенд/БД | **Supabase** (Postgres + Auth + RLS) | Auth, база данных, права доступа |
| Графики | **Recharts** | Диаграммы на дашборде |
| Drag & Drop | **dnd-kit** | Перетаскивание карточек сделок по этапам (воронка) |
| Иконки | **lucide-react** | Иконки (идут с shadcn/ui) |
| Архитектура | **Feature-Sliced Design (FSD)** | Каноничная слоистая архитектура. Демонстрирует заказчику зрелость кода |
| Линтинг | **ESLint** (typescript-eslint) + **Prettier** + контроль границ слоёв FSD | Чистый, консистентный код; сигнал профессионализма |
| Хостинг фронта | **Vercel** | Публичный URL, авто-деплой из GitHub |
| Хранение кода | **GitHub** | Репозиторий + связка с Vercel |

### Требования к качеству кода

- **React 19** — использовать актуальные паттерны (никаких устаревших подходов).
- TypeScript strict: `strict: true` в tsconfig, никаких `any` без явного обоснования.
- Типы для всех данных из Supabase (генерировать через Supabase CLI или описать вручную).
- Компоненты — функциональные, с хуками.
- Данные из Supabase — только через **TanStack Query** (не сырые `useEffect` + `fetch`).
- Формы — только через **React Hook Form + Zod** (не неконтролируемые инпуты).
- Обязательны состояния: **loading**, **empty**, **error** для каждого экрана с данными.
- Адаптивность: приложение должно нормально выглядеть на desktop и mobile.

### Разграничение состояния (важно — не смешивать)

- **TanStack Query** — только серверное состояние: всё, что приходит из Supabase (клиенты, сделки, активности, агрегаты дашборда). Кеш, инвалидация, мутации.
- **Zustand** — только клиентское UI-состояние: открыт/закрыт сайдбар, активные фильтры и поиск, выбранная тема, состояние модалок/диалогов. НЕ дублировать сюда серверные данные.

Это разграничение — сознательное архитектурное решение, его нужно соблюдать по всему проекту.

### Архитектура FSD (границы слоёв)

Соблюдать однонаправленные зависимости между слоями (сверху вниз): `app → pages → widgets → features → entities → shared`. Слой может импортировать только из слоёв ниже себя, не выше и не из соседних слайсов напрямую. Контролировать это через ESLint (см. раздел 8).

### ESLint / Prettier

- `typescript-eslint` (рекомендованные + строгие правила).
- Плагин под React 19 / хуки.
- Контроль границ слоёв FSD (`eslint-plugin-boundaries` или проверка через `steiger`).
- Prettier для форматирования; линт без ошибок — обязательное условие готовности.

## 4. Модель данных (Supabase / PostgreSQL)

Все таблицы в схеме `public`. У каждой пользовательской записи есть `user_id`, ссылающийся на `auth.users`, — это основа изоляции данных через RLS (каждый видит только своё).

### Таблица `profiles`
Расширение стандартной `auth.users` (Supabase хранит логины отдельно).

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid (PK, = auth.users.id) | Идентификатор пользователя |
| `full_name` | text | Имя пользователя |
| `avatar_url` | text (nullable) | Ссылка на аватар (опционально) |
| `created_at` | timestamptz (default now()) | Дата создания |

Профиль создаётся автоматически при регистрации (через триггер в Supabase или в коде после signup).

### Таблица `clients`

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid (PK, default gen_random_uuid()) | ID клиента |
| `user_id` | uuid (FK → auth.users) | Владелец записи |
| `name` | text (not null) | Имя контакта |
| `company` | text (nullable) | Компания |
| `email` | text (nullable) | Email |
| `phone` | text (nullable) | Телефон |
| `status` | text (not null, default 'lead') | Статус: `lead` / `active` / `inactive` |
| `created_at` | timestamptz (default now()) | Дата создания |
| `updated_at` | timestamptz (default now()) | Дата обновления |

### Таблица `deals`

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid (PK) | ID сделки |
| `user_id` | uuid (FK → auth.users) | Владелец |
| `client_id` | uuid (FK → clients) | Связанный клиент |
| `title` | text (not null) | Название сделки |
| `amount` | numeric (default 0) | Сумма сделки |
| `stage` | text (not null, default 'new') | Этап: `new` / `negotiation` / `won` / `lost` |
| `expected_close_date` | date (nullable) | Ожидаемая дата закрытия |
| `created_at` | timestamptz (default now()) | Дата создания |
| `updated_at` | timestamptz (default now()) | Дата обновления |

### Таблица `activities`

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid (PK) | ID активности |
| `user_id` | uuid (FK → auth.users) | Владелец |
| `client_id` | uuid (FK → clients) | Связанный клиент |
| `type` | text (not null) | Тип: `call` / `email` / `meeting` / `note` |
| `content` | text (not null) | Текст активности |
| `created_at` | timestamptz (default now()) | Дата создания |

### Связи
- `clients.user_id` → `auth.users.id`
- `deals.client_id` → `clients.id` (ON DELETE CASCADE)
- `activities.client_id` → `clients.id` (ON DELETE CASCADE)

### Row-Level Security (RLS) — ОБЯЗАТЕЛЬНО

Включить RLS на всех таблицах (`clients`, `deals`, `activities`, `profiles`). Политики:

- **SELECT / INSERT / UPDATE / DELETE** разрешены только если `user_id = auth.uid()`.
- Для `profiles`: пользователь читает/редактирует только `id = auth.uid()`.

Это критично: демонстрирует заказчику, что разработчик понимает безопасность данных (RLS — то, что явно спрашивают в серьёзных заказах, особенно health/fintech).

## 6. Функциональные требования по модулям

### 6.1. Авторизация (Auth)

**Регистрация (`/register`):**
- Поля: имя, email, пароль (форма на React Hook Form + Zod, валидация email/длины пароля).
- При успехе: создаётся пользователь в Supabase Auth + запись в `profiles`, редирект на дашборд.
- Показывать ошибки (email занят, слабый пароль).

**Вход (`/login`):**
- Поля: email, пароль.
- При успехе: редирект на дашборд.
- Обработка ошибки (неверные данные).

**Сессия:**
- Хранить сессию Supabase, восстанавливать при перезагрузке страницы.
- Защищённые роуты редиректят на `/login`, если нет сессии.
- Кнопка «Выход» (signOut) в шапке/меню.

**Демо-доступ (важно для портфолио):**
- Сделать заранее заготовленный демо-аккаунт (email/пароль) с наполненными данными, ИЛИ кнопку «Login as demo» на странице логина.
- Причина: заказчик, открывший ссылку, должен сразу увидеть наполненный интерфейс, а не пустую форму регистрации. Это критично для впечатления.

### 6.2. Клиенты (`/clients`)

- **Таблица** клиентов: колонки — имя, компания, email, статус (бейдж с цветом), дата создания.
- **Поиск** по имени/компании (input сверху, фильтрация).
- **Фильтр** по статусу (lead / active / inactive).
- **Кнопка «Добавить клиента»** → диалог (shadcn Dialog) с формой (React Hook Form + Zod).
- **Редактирование** клиента (тот же диалог, предзаполненный).
- **Удаление** клиента (с подтверждением; каскадно удаляет его сделки и активности).
- Клик по строке → переход на `/clients/:id`.
- Состояния: loading (скелетон), empty («Нет клиентов, добавьте первого»), error.

### 6.3. Карточка клиента (`/clients/:id`)

- Блок с инфо о клиенте (имя, компания, email, телефон, статус) + кнопка «Редактировать».
- **Сделки клиента** — список связанных сделок (название, сумма, этап).
- **Лента активностей** (timeline) — активности клиента по дате (тип-иконка + текст + дата).
- **Кнопка «Добавить активность»** → форма (тип + текст).
- Состояния: loading, empty (для сделок и активностей отдельно), error.

### 6.4. Воронка сделок (`/deals`)

- **Kanban-доска**: 4 колонки по этапам — New / Negotiation / Won / Lost.
- Карточка сделки: название, клиент, сумма.
- **Drag & drop** карточек между колонками (dnd-kit) → при перетаскивании обновляется `stage` в Supabase.
- Сумма по каждой колонке (итог сверху колонки).
- **Кнопка «Добавить сделку»** → диалог (название, клиент из выпадающего списка, сумма, этап, дата).
- Состояния: loading, empty, error.

> Если drag & drop окажется сложным на первом заходе — допустимо на первом этапе сделать сделки простым списком/таблицей с выпадающим выбором этапа, а Kanban с dnd-kit добавить в фазе полировки. Но целевой вид — именно Kanban (это эффектно на живой ссылке).

### 6.5. Дашборд (`/dashboard`)

Главный экран, первое, что видит заказчик. Должен выглядеть максимально «продуктово».

**Карточки метрик (KPI, сверху):**
- Всего клиентов
- Активных клиентов
- Сумма открытых сделок (new + negotiation)
- Сумма выигранных сделок (won)

**Графики (Recharts):**
- **Воронка/бар-чарт**: количество сделок по этапам.
- **Пай-чарт**: клиенты по статусам (lead/active/inactive).
- **Линейный график**: новые клиенты по месяцам (динамика).

**Лента последних активностей** (последние 5-10 записей из `activities` по всем клиентам).

Состояния: loading (скелетоны карточек и графиков), empty (если данных нет), error.

## 8. Структура проекта — Feature-Sliced Design (FSD)

Проект построен по каноничной методологии **FSD**. Слои (сверху вниз по уровню ответственности): `app → pages → widgets → features → entities → shared`. Зависимости строго однонаправленные: слой импортирует только из нижележащих слоёв. Каждый слайс инкапсулирован и раскрывает наружу только свой публичный API (`index.ts`).

```
src/
  app/                         # Инициализация приложения
    providers/                 #   QueryClientProvider, RouterProvider, ThemeProvider
    styles/                    #   глобальные стили, Tailwind
    index.tsx                  #   точка сборки провайдеров

  pages/                       # Страницы (композиция виджетов под роут TanStack Router)
    dashboard/
    clients-list/
    client-detail/
    deals/
    login/
    register/

  widgets/                     # Крупные самостоятельные блоки UI
    app-sidebar/               #   боковое меню навигации
    app-header/                #   шапка (пользователь, выход)
    dashboard-kpi/             #   блок KPI-карточек
    dashboard-charts/          #   блок графиков
    clients-table/             #   таблица клиентов с поиском/фильтром
    deals-board/               #   Kanban-доска сделок
    client-activity-timeline/  #   лента активностей клиента

  features/                    # Действия пользователя (сценарии с мутациями)
    auth-login/                #   форма входа
    auth-register/             #   форма регистрации
    auth-logout/               #   выход
    client-create-edit/        #   диалог создания/редактирования клиента
    client-delete/             #   удаление клиента
    deal-create-edit/          #   диалог сделки
    deal-change-stage/         #   drag&drop смена этапа (dnd-kit + мутация)
    activity-add/              #   добавление активности

  entities/                    # Бизнес-сущности (модель + базовый UI + запросы)
    client/                    #   тип, ClientCard, useClientsQuery, useClientQuery
    deal/                      #   тип, DealCard, useDealsQuery
    activity/                  #   тип, ActivityItem, useActivitiesQuery
    user/                      #   профиль, сессия, useSession

  shared/                      # Переиспользуемое, без бизнес-логики
    api/                       #   supabase.ts (клиент), сгенерированные типы БД
    ui/                        #   обёртки shadcn/ui, EmptyState, ErrorState, Skeleton
    lib/                       #   утилиты, хелперы, форматтеры
    config/                    #   константы, env
    store/                     #   Zustand-сторы UI-состояния (sidebar, filters, theme)
```

**Пояснения по слоям для реализации:**

- **entities** держат «данные и их отображение» — типы сущностей, query-хуки (TanStack Query для чтения), базовые презентационные компоненты (карточка клиента без логики действий).
- **features** держат «действия над данными» — мутации (создать/изменить/удалить), формы, диалоги. Именно тут React Hook Form + Zod и мутации TanStack Query.
- **widgets** собирают entities + features в законченные блоки (таблица клиентов = entities/client + features/client-create-edit + client-delete).
- **pages** собирают widgets под конкретный роут.
- **shared/store** — здесь живёт Zustand (только UI-состояние), **shared/api** — Supabase-клиент и типы БД.

Соблюдение границ между слоями контролируется ESLint (раздел «Требования к качеству кода»).
